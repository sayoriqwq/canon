# Concept

## Markdown 文档即模块

Markdown 文档应被视为可组合的模块单元。

文档不是散乱文本容器，而是最小组织单元。它应该可以被索引、链接、组合、复用，并在后续维护中保持稳定边界。

## 文档构成

文档由 metadata 和结构化正文组成。

metadata 承载文档级别的持久化状态信息。结构化正文承载可阅读、可维护、可迭代的内容。

## Metadata 层

文档 metadata 用来描述文档的读者、生成来源、review 状态和使用目的。

`audience` 只描述读者类型，只使用 `agent`、`human` 或二者列表。不要把具体人名、任务描述、场景说明塞进 `audience`。

`authors` 描述文档由谁生成或主要撰写。例如由 codex 写，则标记 `codex`。

`reviewed_by` 描述谁明确 review 或批准过该文档。若 sayori 已明确通过，则标记 `sayori`。

`purpose` 用一句话说明这份文档帮助读者完成什么。

metadata 不应承载正文论述。正文中需要展开的概念、理由、规则，应放入结构化正文。

## 文档位置和命名

目录和文件名是查询规范。

文档位置和命名应帮助 agent 快速定位目标文档。命名需要表达文档职责，不要只表达临时状态或作者习惯。

如果一个文档的位置或名字不能帮助查询，它就会增加后续维护成本。

## 受众和使用场景优先

文档不是先写内容，再反推读者。先明确读者是谁，以及读者打开这份文档要完成什么。

读者和使用场景决定文档的结构、语气、细节密度和边界。

## 单一职责

一个文档应承担一个主要职责。

不要把语义背景、操作规则、历史决策、索引导航、当前偏好混在同一个文档里。相关内容可以互相链接，但不要互相吞并。

单一职责不是追求文件数量越多越好，而是让每个文档成为稳定、可组合、可复用的组织单元。

## 结构会训练未来维护行为

如果文档结构会影响后续维护方式，就应从一开始使用目标结构，而不是因为当前内容少就临时合并。

对需要长期维护的文档系统，偏好拆分文件和拆分 index。不要因为系统暂时还小，就保留一个巨大的 index。

## 结构化正文

正文应该结构化，而不是完全依赖模型自由发挥。

结构化正文让后续维护有稳定入口，也让同类文档可以持续迭代。标题、段落职责、列表形态和必要字段应该尽量稳定。

对于需要长期复用和维护的文档类型，优先用脚本约束结构，而不是只靠 prompt 或模型自觉。

脚本不属于 profile 的必备实现，但这是跨目录写作偏好：当文档结构需要长期稳定时，应该考虑用脚本提供可检查的结构约束。
